## 1. 文档定位

1. 本文档是对当前执行规范的“审核草案”，用于先评审后落地。
2. 在你批准前，不修改 `CLAUDE.md` 规范正文。
3. 批准后，将本文内容并入 `CLAUDE.md`，并以 `CLAUDE.md` 作为唯一执行约束，不再引入新的并行规范文档。

## 2. 目标与原则

1. 优先保证可复用性、可扩展性、稳定性，再考虑最小改动成本。
2. 业务模块与基础模块分层清晰，尤其保持播放器核心模块边界稳定与独立。
3. 新增功能、重构、修复 Bug 统一遵循同一流程，避免“临时修补”长期堆积。

## 3. 执行优先级（统一口径）

1. 正确性与稳定性（不能引入回归或破坏关键路径）。
2. 模块边界与可复用性（组件抽象、职责清晰、避免复制逻辑）。
3. 可扩展性（类型、路由、筛选、配置可增量扩展）。
4. 一致性（交互、样式、组件使用统一）。
5. 最小改动（在满足以上 1-4 后，控制改动范围）。

## 4. 开发流程（新组件 / 新功能 / Bug 修复）

1. 先确认边界与契约：输入输出、状态归属、依赖方向、与现有模块关系。
2. 优先复用或扩展已有共享组件；如需新建，先定义可复用接口再落地。
3. 避免在业务页面堆积 UI 状态与逻辑，复杂行为下沉到可测试模块。
4. 改动后至少执行针对性验证（类型检查、lint、相关单测/集成测试）。
5. 提交按功能原子化（一个目标一组改动），保证可回滚与可审查。

## 5. 播放器模块专项约束

1. 播放器核心层（core）优先保持独立，不把页面级临时逻辑直接写入核心。
2. 播放器壳层（shell）负责页面布局、外部交互、业务编排。
3. 视觉与交互统一由主题令牌与共享组件驱动，避免硬编码与样式漂移。
4. 涉及断点播放、线路切换、影院模式等关键路径时，必须做回归验证。

## 6. 风险分析（现状执行策略）

1. 过度强调“最小改动”：
   - 风险：重复实现、边界侵蚀、后续维护成本升高。
   - 对策：将“最小改动”降级为后置约束，先满足架构与稳定性。
2. 多处散落规范并行：
   - 风险：冲突口径导致执行不一致。
   - 对策：以 `CLAUDE.md` 汇总收敛，其他文档仅保留任务/方案记录属性。
3. 修复导向但缺少抽象沉淀：
   - 风险：短期可用，长期扩展困难。
   - 对策：每次修复判断是否应沉淀到共享层（组件、hooks、服务）。
4. 验证覆盖不足：
   - 风险：交互类问题（层级、遮挡、点击穿透）反复出现。
   - 对策：关键路径建立最小可行回归集（单测 + 关键页面冒烟）。

## 7. 审核后落地方式

1. 你确认本文后，我将把可执行条款迁入 `CLAUDE.md`（同日更新）。
2. 更新后本文档可标记为“已合并”或删除，避免双规范并存。
3. 后续如再新增执行口径，直接改 `CLAUDE.md`，不新增平行规范文件。
