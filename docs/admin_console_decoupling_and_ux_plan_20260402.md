# Resovo — 后台控制台解耦与交互改造方案（2026-04-02）

> status: active
> scope: admin console only
> reviewed_by: AI engineering review
> decisions_confirmed: 2026-04-02

## 1. 文档目标

在既有《前后台彻底解耦实施方案》基础上，补充后台页面级交互重构与数据模型约束，解决当前“功能可用但体验不一致、状态语义不清、批量治理效率低”的问题。  
本文件仅用于评审和冻结方案，不包含代码变更。

---

## 2. 设计原则（本轮统一标准）

1. 后台是独立控制面板，交互优先级是”高效率+低歧义+可追溯”。
2. 同类操作在不同页面必须使用同一交互范式（批量栏、消息反馈、状态控件）。
3. 状态展示必须与数据库语义一一对应，禁止”UI 二值控件表达三态逻辑”。
4. 所有管理动作反馈不挤压页面布局，统一右下角临时浮窗。
5. 先统一信息架构与治理链路，再做视觉与微交互优化。
6. **⚠️ 审核补充**：`review_status × visibility_status × is_published` 状态机矩阵（合法组合表 + 每个跃迁是 UI 拦截还是 API 强制）为 Phase 0 必须交付物，是 Phase 2 视频管理重构的编码依据，不得推迟。

---

## 3. 当前问题归纳（评审结论）

### 3.1 批量操作栏位置不一致

1. 采集控制台的批量操作在表格上方（inline）。
2. 视频/播放源等表格在底部悬浮（sticky-bottom）。
3. 结果：用户切换页面后心智模型断裂，操作定位成本高。

### 3.2 采集源排序不符合直觉

1. 采集源接口默认按 weight/key 排序。
2. 下拉筛选直接复用接口顺序，用户看到“顺序混乱”。
3. 结果：筛选效率下降，易误选来源。

### 3.3 内容审核页面能力不完整

1. 待审核列表缺过滤、排序、批量审核。
2. 局部字段显示存在 undefined/不一致映射风险。
3. 审核详情播放器缺“播放源/选集”切换，视频失败无回退。

### 3.4 视频管理状态模型与控件冲突

1. 可见性是三态（public/internal/hidden），当前使用 toggle（适合二态）。
2. “可见性/审核状态/上架状态”关系在页面层未被明确解释。
3. 操作菜单复杂（快速编辑、完整编辑、上下架、同步并列），信息架构冗余。

### 3.5 播放源管理视图粒度不合理

1. 当前“一个源 URL 一行”，治理链路分散。
2. 无法快速从“某站点失效”联动到受影响视频主体集合。
3. 站点级、视频级、视频+站点级治理视角未形成闭环。

### 3.6 用户管理能力缺口

1. 缺用户删除能力（建议软删除）。
2. 内容质量表未与统一表格体系对齐，缺列设置/一致操作。

### 3.7 消息提示机制不一致（新增补充）

1. 有的提示出现在表格内，有的在表格外，有的在工具栏旁。
2. 仍有浏览器 `alert()` 弹窗。
3. 结果：视觉噪音高，反馈位置不可预测，且影响布局稳定性。

## 4. 目标架构（后台交互层）

## 4.1 统一消息中心（Admin Toast）

1. 后台全局挂载 `AdminToastHost`，位置固定右下角。
2. 所有”操作反馈”统一走 `notify.success/info/warn/error`。
3. 默认策略：
   - success/info: 2.5-3.5s 自动关闭
   - warn/error: 5-8s 自动关闭，可手动关闭
   - 同类消息按 `dedupeKey` 去重合并
   - 同屏最多展示 3 条，超出进入队列
4. 视觉要求：紧凑、不占文档流、不可推动表格布局。
5. 保留内联反馈边界：字段校验错误继续内联；阻断级错误保留页面级 Alert。

**⚠️ 审核补充 — 反馈类型→控件选择决策表（Phase 0 冻结，Phase 1 执行依据）**

| 反馈类型 | 控件 | 说明 |
|----------|------|------|
| 操作成功（增/改/删/验证通过） | Toast success | 右下角，2.5-3.5s 自动关闭 |
| 操作失败（API 错误、网络超时） | Toast error | 右下角，5-8s，可手动关闭 |
| 操作警告（部分成功、有风险） | Toast warn | 右下角，5-8s，可手动关闭 |
| 字段校验错误 | 内联（input 下方） | 不走 toast，就近显示 |
| 阻断级错误（权限不足、会话失效） | 页面级 Alert | 占位布局，需用户主动确认 |
| 系统级持久告警（服务异常） | Toast + 持久 | 不自动关闭，队列最前 |
| 浏览器 `alert()` | ❌ 禁止 | 全部替换为上述方案 |

## 4.2 统一批量操作栏

1. 后台所有列表页统一使用底部悬浮批量栏（sticky-bottom）。
2. 顶部 inline 模式仅允许在“非表格区域批处理器”场景使用。
3. 统一包含：选中计数、主操作组、取消选择、loading/禁用态。

## 4.3 统一状态表达

1. 三态字段一律使用三选一控件（Segmented/Select），禁止 Toggle。
2. 二态字段可用 Toggle，但必须有文本标签与后端状态一致映射。
3. 涉及审核/发布/可见性的动作必须显示状态约束提示。

## 5. 页面级改造方案

## 5.1 采集控制台

1. 批量栏从顶部 inline 收敛为底部悬浮，与视频/源管理一致。
2. 保留高级筛选区，但不承载批量反馈提示。
3. 触发采集、批量变更、导入导出反馈统一走右下角 toast。

## 5.2 视频管理（重点重构）

### 5.2.1 操作区重构

1. 移除”操作下拉菜单”。
2. 合并”快速编辑 + 完整编辑”为单一”编辑”入口（图标按钮）。
3. 行内操作仅保留高频动作：
   - 编辑（铅笔图标）
   - 前台详情（外链图标）
   - 前台播放（播放图标）
   - 上架/下架（Toggle，二态）
4. **⚠️ 审核补充 — 上线策略（已确认：灰度开关）**：
   - Phase 2 上线时通过 `localStorage` feature flag（key: `admin_video_ops_v2`）区分新旧操作区
   - 内部验证通过后（约 2 周），全量切换并强制移除该 flag
   - 目的：降低操作习惯断裂风险，避免长期双版本维护

### 5.2.2 编辑面板重构

1. 编辑面板采用统一抽屉/侧栏，不再区分“快速/完整”。
2. 分区：
   - 基础元数据（标题、年份、类型、题材、地区、简介等）
   - 源信息摘要（活跃源/总源、异常标记）
   - 元数据同步（豆瓣同步）
3. 豆瓣同步并入编辑流程，提供“预览变更/选择应用字段/确认应用”。

### 5.2.3 类型体系统一

1. 所有编辑/筛选/展示入口统一使用完整 `VideoType` 枚举：
   - movie, series, anime, variety, documentary, short, sports, music, news, kids, other
2. 统一 label map，禁止页面私有枚举分叉。

### 5.2.4 状态关系明确化（页面内可见说明）

1. `review_status`：治理流程状态（pending_review/approved/rejected）
2. `visibility_status`：曝光策略（public/internal/hidden）
3. `is_published`：上架开关（前台可播放入口）
4. 建议交互约束：
   - rejected 不允许直接上架（需先改审核状态）
   - hidden/internal 上架时给出约束提示
   - 任何自动联动需在 UI 文案显式说明

## 5.3 内容审核

1. 待审列表补齐过滤（类型/来源/时间）和排序（提交时间/风险级别等）。
2. 增加批量审核（批量通过/拒绝，拒绝原因模板）。
3. 审核详情播放器支持：
   - 多播放源切换
   - 选集切换
   - 失败回退（自动尝试下一源）

## 5.4 播放源管理（三视图）

1. 视图 A：按源站（site）  
   - 一行一个源站  
   - 动作：站点连通性验证
2. 视图 B：按视频主体（video）  
   - 一行一个视频主体  
   - 可展开查看该视频下全部源
3. 视图 C：按“视频主体 + 源站”聚合  
   - 一行一个 video+site 组合  
   - 可展开查看该组合下分集源  
   - 验证动作作用于该组合内全部分集

联动规则：
1. 站点验证失败可筛出该站点关联的全部视频主体与组合。
2. 某 URL 失效可反查同视频主体下其他源，以及该 URL 所属站点集合。
3. 手动验证与自动巡检共用同一任务轨道与结果模型。

## 5.5 用户管理

1. 增加用户删除能力（软删除 + 可审计）。
2. 用户操作栏补齐删除动作（权限与二次确认）。
3. “内容质量表”迁移到共用 ModernDataTable，补齐列设置、筛选、排序能力。

## 6. 数据与接口调整（为交互提供支撑）

1. 采集源接口支持排序参数：
   - `sortBy=name|weight`
   - 下拉默认 `name`，管理页可切换为 `weight`
2. 审核列表接口扩展过滤/排序参数与批量审核接口。
3. 视频编辑接口对齐全量 `VideoType` 约束与统一字典。
4. ~~播放源域建议新增 `video_sources.site_key`（或等价字段）并回填，用于三视图稳定联动。~~ **⚠️ 审核修订（已撤销）**：经数据结构分析，`video_sources → videos.site_key → crawler_sites` JOIN 链路已完整存在，现有索引支撑三视图聚合查询。在 `video_sources` 冗余存储 `site_key` 会引入双写一致性问题，且语义错误（站点归属是视频级属性）。**Phase 3 无需任何 schema 变更，直接基于现有 JOIN 实现三视图。**
5. 消息中心无需后端改造，但前端需替换所有局部 toast/alert 接线。

## 7. 分阶段实施（建议）

## Phase 0 — 规范冻结（当前阶段）

1. 冻结交互规范、状态机说明、字段字典。
2. 冻结消息提示规范与页面迁移优先级。

## Phase 1 — 低风险一致性收敛

1. 全局 `AdminToastHost` 上线，替换 `alert()`。
2. 批量栏统一到底部悬浮。
3. 采集源下拉排序改为名称优先。

## Phase 2 — 核心页面重构

1. 视频管理操作区重构（去下拉，合并编辑，接入跳转）。
2. 视频编辑面板整合同步预览/应用流程。
3. 审核页补齐过滤、排序、批量审核与多源播放器。

## Phase 3 — 播放源治理升级

1. ~~数据模型补齐（site_key 回填）。~~ **已撤销**，无需 schema 变更。
2. 三视图与联动筛选上线（基于现有 `video_sources JOIN videos.site_key` 实现）。
3. 自动巡检流程接入并提供运营反馈闭环。

## 8. 验收标准（DoD）

1. 后台所有操作反馈均出现在右下角全局 toast，页面无局部漂移提示。
2. 后台不存在浏览器 `alert()`。
3. 视频管理无”快速编辑/完整编辑”双入口，操作区无下拉菜单。
4. 视频类型在筛选、表格、编辑页面显示与选项完全一致。
5. 审核页面具备过滤、排序、批量审核、多源选集播放能力。
6. 播放源管理可在站点/视频/视频+站点三个维度完成验证与联动排查（基于现有 JOIN，无需 schema 变更）。
7. 用户管理具备删除能力；内容质量表具备统一表格能力。

---

## 9. 风险与注意事项

1. `visibility_status` 与 `is_published` 历史兼容逻辑复杂，重构前必须先冻结状态机文档。
2. 播放源三视图若缺 `site_key` 将导致联动精度不足，需优先补齐模型。
3. 视频编辑流程合并后，需严格保留权限边界（如豆瓣同步 admin only）。
4. 所有重构必须保持 API 向后兼容或提供迁移窗口，避免影响现有后台自动化脚本。
