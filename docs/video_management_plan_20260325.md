# 视频治理与管理方案 (2026-03-25)

## 方案背景
后台管理已经基本完成采集，预合并工作，基于当前系统已经设计好的 `content_governance_schema_plan.md` 规范与 `video_sources` 的数据库结构逻辑，结合流媒体平台标准的业务实践，我们在不动代码的基础上，设计如下视频管理方案。整个方案分为「视频管理业务流转架构」与「对应的后台管理页面布局」两部分。

## 一、 视频管理业务流转架构

基于现状，管理底层完全依赖于两大核心维度的状态字段组合：
1. **视频本身治理（基于 `videos` 表）**：
   * **`review_status`审核状态**：`pending_review` (待审), `approved` (通过), `rejected` (拒绝), `blocked` (封锁)
   * **`visibility_status`可见性**：`public` (公开), `hidden` (隐藏/下架), `internal` (内部/入库), `blocked` (封锁)
2. **播放源维度（基于 `video_sources` 表）**：
   * **`is_active`**：源链接是否存活。
   * **`last_checked`**：上一次检测连通性的时间。

在这个底层架构上，定义如下 4 个维度的标准工作流：

### 1. 采集入库路由策略 (Ingestion Routing)
* 根据 `crawler_sites.ingest_policy` 决断：
  * **高信赖源 (如大站采集)**：若 `allow_auto_publish=true`，落库直接为 `review=approved`, `visibility=public`。
  * **非信赖源 (易含违规词或画质差)**：`allow_auto_publish=false` 时，落库默认为 `review=pending_review`, `visibility=internal`。等待人工审核。

### 2. 内容审核工作流 (Review Workflow)
* 管理员查阅库中 `pending_review` 的视频。
* **通过**：更新视频为 `review_status = 'approved'`，并同步标记 `visibility_status = 'public'` 完成上架。
* **驳回/封禁**：如果是内容违规或黑产广告等，标记为 `rejected`/`blocked`，并自动设为 `hidden`/`blocked` 避免透出。

### 3. 上下架管理 (Publish / Unpublish)
* **下架操作不等于打回重审**：已上线的视频，如果不涉及违规（仅仅是暂时无版权或运营策略调整），单独将 `visibility_status` 从 `public` 改为 `hidden`；`review_status` 仍然是 `approved`。未来随时可一键重新变成 `public`（上架）。

### 4. 视频源可用性健康验证 (Source Verification)
* **定时/异步检测任务**：通过后台 Worker 扫表或在后台面板提供“批量触发源检测”的操作，检查各个 `video_sources` 的 `source_url` 的 Http 状态。
* **状态联动 (推荐策略)**：如果一个已上架的视频，其所有对应的源都被标为 `is_active = false`（全部死链），后台应产生异常告警。运营人员可决策是否暂时把它 `visibility_status` 调整为 `hidden`（即下架），以防止 C 端用户点开无法播放导致的客诉。

## 二、 后台管理页面布局模块设计

基于上述工作流系统，后台菜单和页面分为 3 个核心界面：

### 界面一：内容审核台 (Moderation Dashboard)
> 定位：高效率处理新增的、待处理的任务（类似流水线作业）。

* **顶部数据统计板**：待审核视频数、今日已审数、违规拦截率。
* **页面布局（左右 / 抽屉式布局）**：
  * **左侧列表区**：瀑布流或紧凑列表，仅展示 `pending_review` 的视频。包含：海报图、标题、采集来源站、分类（`normalized_type`）和采集时间。
  * **右侧/侧边抽屉 (评审详情区)**：点击某一条目，拉出面板。
    * **内容预览**：直接展示抓取到的视频 Title、描述（用于查阅是否含违规词/垃圾广告）。
    * **快速播放测试**：提供一个小型播放器界面获取 `video_sources` 第一条源进行播放。
    * **操作按钮组**：`[ 🟢 通过并上架 ]`、`[ 🔴 拒绝 (内容低劣) ]`、`[ 🚫 封禁 (违禁) ]`。支持快捷键。

### 界面二：全量视频治理库 (Video Inventory Management)
> 定位：日常对所有影视作品进行宏观管理、排查、上下架控制。

* **顶部综合筛选栏**：
  * 搜索组件（标题/ID），筛选类型（电影/剧集）。
  * **状态组合筛选器**：
    * 上下架状态：仅看公开(Public) / 仅看下架隐藏(Hidden)。
    * 审核状态：已通过 / 已拒绝等。
* **数据表格展示 (Data Table)**：
  * 基础信息（封面缩略图 + 主标题 + 短 ID）。
  * 分类与格式（单集电影/多集剧集）。
  * 播放源健康度（微件：比如打一个 Tag：`🟢 3 活跃源` 或 `🔴 源全部失效`）。
  * 上架状态（Switch 开关展示当前 `visibility_status`，管理员可直接快捷上下架）。
  * 操作项（`[编辑]` `[源管理]`)。
* **详情编辑模态框**：直接管理视频本身的元数据，并独立查阅「分集/播放源管理」子面板。

### 界面三：视频源与健康度监控中心 (Source Health Center)
> 定位：解决死链、爬虫失效、源不可用导致的用户体验灾难。

* **告警提示横幅 (Alert Bar)**：提示因所有源失效而产生“空壳”的已上架视频数量，提供批量下架按钮。
* **双 Tab 工作区**：
  * **Tab 1：失效源管理 (Dead Sources)**
    * 列表展示所有 `is_active = false` 的物理源。
    * 显示关联的视频名称、第几季第几集、原 `source_url`、最后检测时间 (`last_checked`)。
    * 操作：提供 `[重新检测]`、`[彻底删除 (Delete)]`，或手动替换新的直链 URL。
  * **Tab 2：用户纠错与报错 (User Submissions/Reports)**
    * 展示用户投稿的新源或者来自于用户的死链报错（关联 `submitted_by IS NOT NULL AND is_active = false`）。
    * 操作：`[采纳为有效源 (Approve)]` / `[忽略报错 (Reject)]`。

## 三、 高阶运维保障与进阶防御机制

在实际大规模流媒体管理中，为了支撑千万级数据和高效人工干预，除了上述的核心工作流外，还需要以下高级机制（根据优先级分为现阶段与未来规划）：

### 现阶段计划 (P1)

#### 1. 操作审计与撤销/重做机制 (Audit Logs & Undo/Redo)
建立 `operation_logs` 表，记录核心更新（旧状态/新状态快照）。针对管理后台进行的所有更改（批量下架、修改元数据等），支持追溯与撤销。对于复杂合并删除的 Undo 需依赖数据表层面的“软删除 (Soft Delete)”。

#### 2. “局部保护锁” (Field-Level Locking)
引入 `locked_fields` 元数据层，保障人工精心修订过的标题、简介、标签等不被下一周期的定时爬虫重新覆盖刷新。

#### 3. “查重”与“实体合并/拆分” (Manual Merge & Split)
允许将重复抓取或异源的同名影片关联实体强行合并。将重复的 `video_sources` 重新分配给主键视频，并通过重定向（Redirect）或替换，确保展示唯一权威源数据。

### 未来任务规划 (P2/P3 暂不在现阶段开发计划内)

#### 4. 全局黑名单与特征自动化拦截 (Global Anti-Spam / Blocklists)
针对固定包含博彩/低俗水印特征的域名或特定正则表达式，建立全局过滤系统，使爬虫在此阶段直接判定为黑名单抛弃不入库，削减极大人工审核成本。

#### 5. 针对版权法务诉讼的强隔离 (DMCA / Copyright Takedowns)
引入法务维权/版权投诉相关下架状态。与普通的 `hidden` 不同，需关联法务投诉编号。此状态禁止任何低权限人员或自动化采集工具将视频再次置于 `public`。

#### 6. 审核多级提权与复核机制 (Escalation & Secondary Review)
按角色权限 (RBAC) 拆分审核深度。初级审核员遇到争议视频，能且只能发起 `[升级处理 (Escalate)]`；最终的上架或终结许可交由高级管理员审批复合池进行二次判决。
