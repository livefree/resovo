## 2. 黄线项（必须落地或显式登记）

### Y1 — primitive gray ramp 实装值与方案文档 / ADR 承诺值漂移

**事实**：`primitives/color.ts` 落地的 gray ramp dark 段五档 L 值与方案文档 §4.1 + ADR-111 §决策第 1 条最初承诺值系统性漂移 +1.5~2 百分点：

| step | 文档承诺 L | 实装 L | 差距 | 设计 hex（CHG-UI-02a 校准目标）|
|---|---|---|---|---|
| 1000 | 6.5% | **8.0%** | +1.5% | `#0b0d10` |
| 950 | 11.2% | **12.0%** | +0.8% | `#11141a` |
| 925 | 13.5% | **15.0%** | +1.5% | `#161a22` |
| 900 | 16.5% | **18.0%** | +1.5% | `#1d222c` |
| 800 | 23.0% | **21.0%** | -2.0% | `#252b37` |

**根因**：CHG-UI-02a 用户截图反馈触发的"OKLCH → sRGB 校准"之后，实装值已是用户视觉验收的最终值，但方案文档 §4.1 + ADR-111 §决策第 1 条未同步回填。

**风险**：未来回看 ADR / 方案文档时无法溯源数值依据，下一批密度 token 微调时无锚点。

**处置**（CHG-UI-06 落地）：
- 方案文档 §4.1 同步标注"02a 校准后实装值"
- ADR-111 §决策第 1 条同步标注 02a 增量
- 不重新落地代码（实装值已被用户验收）

### Y2 — 缺自动化"OKLCH → 设计 hex"对齐测试

**事实**：`semantic.test.ts` `primitive reference integrity` 部分用 PRIMITIVE_VALUES 集合 + color-mix/linear-gradient 白名单约束硬编码——这是阻断硬编码 hex 的强结构防线，但**缺一个"primitive ramp L/C/H → 估算 sRGB hex" 的对齐快照单测**。

**风险**：CHG-UI-02a 校准基本靠人眼+用户反馈，下次 ramp 微调（第二批密度 / 第三批 chip）若误改 ramp 将无法被自动捕获。

**处置**：记入下批 token 序列（不在本批范围）。
